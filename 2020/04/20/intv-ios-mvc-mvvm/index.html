MVC和MVVM差異

Posted on 2020-04-20 | In 面試考題 |

這系列筆記，替自己紀錄iOS面試常問考題。

MVC也是APPLE官方推薦的設計模式，Model負責資料、View負責畫面呈現、Controller負責View跟Model數據傳遞。
MVVM解決MVC中，ViewController過度擁腫問題，把邏輯移動到ViewModel，把轉場動畫留在View，ViewModel與View需要做資料綁定。

MVC

MVC也是APPLE官方推薦的設計模式，Model負責資料、View負責畫面呈現、Controller負責View跟Model數據傳遞。
隨著程式開發，會發現很多邏輯的東西都會放在VC裡面，導致VC變得臃腫(Massive View Controller)。

好處：

  1. 提供結構，增強程式的可維護性
  2. 程式碼重用，在MVC設計模式中，同一個 Model 可以被不同的Controller重複使用
  3. 工作分配，多個工程師可以同時編寫不同部分的程式碼

缺點：

  1. 需事前嚴謹規劃
  2. 用於小專案成本高

MVVM

因為Cocoa的ViewController實在太像View了，所以在MVC的V改成View跟ViewController，C由ViewModel取代，已就變成MVVM了，ViewModel負責串接Model，所以ViewModel就處理『負責邏輯相關』，如錯誤處理，跟『呈現的邏輯』(這裡說的是presentation，把Model資料整理要放到View上呈現的資料格式)，View就專心處理動畫、轉場…等，ViewModel與View之間需要做資料綁定，常見做法是用Closure或是RxSwift。

好處：

  1. 易於測試
  2. 低耦合，ViewModel可以被不同View使用

缺點：

  1. Controller跟Presentation layer定義模糊
  2. 缺少router跟builder，於是有VIPER、MVVM-C、MVVM+FlowController等方式優化

自己的經驗

覺得pattern這種東西真的是多寫多經驗，我把自己的side project從MVC改成MVVM，雖然沒有完全改過去，有些礙於功力不足還不能切割很完美，也算是有一個初體驗，以下紀錄一下我做了哪些事情：

  1. Group重新規劃：雖然這跟MVVM沒關係，但原本資料夾以MVC去區分，隨著功能變多，變得雜亂，最後以Feature去做區分
  2. Storyboard切分：把一整個Main Storyboard切分為數個
  3. 把coredata獨立出來：操作core data的程式都獨立出來
  4. 建立ViewModel，把cv變成單純的view，邏輯都搬到ViewModel，比如說要讀取Model到呈現在View，中間整理資料流的code
  5. Table cell也清理，把cell中的邏輯搬到model處理，cell就會很單純呈現view的工作，參考 【译】处理 iOS 中复杂的 Table Views 并保持优雅 - 掘金
  6. Table selection count, cell count全部由ViewModel提供
  7. 設定closure，包括reload, delete

但當我讀到歡迎來到真實世界 - 原來是那個傳說中的MVVM阿這篇文章，發現View跟ViewModel要Bind啊！才發覺自己的MVVM只有半套，目前作法是View擁有一個ViewModel，藉由操作ViewModel來控制資料流，不過現在想想這樣也滿違反View的功能？待我在研究一番在更新Blog
